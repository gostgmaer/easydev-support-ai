## Fixed during this audit pass

These were found and fixed immediately, not just logged — all in `multi-tannet-auth-services`. One of them (RR-31) was in code written earlier in this same engagement.

| ID | Category | Description | Fix applied |
|----|----------|-------------|--------------|
| RR-31 | Authorization / Cross-Tenant | `GET /users/tenant-admins?tenantId=X` took `tenantId` from a query param with **zero ownership validation** — any tenant's admin could enumerate another tenant's admin emails by changing the query param. This endpoint was built earlier this engagement (for quota-overage email notifications) and shipped with this bug. | `PermissionsGuard`'s tenant-isolation check (`permissions.guard.ts`) extended to inspect `request.query?.tenantId` in addition to `request.params?.tenantId`, with an explicit carve-out for global/non-tenant-scoped service callers (`user.tenantId` falsy) so legitimate cross-tenant service keys aren't broken. |
| RR-32 | Authorization / Cross-Tenant | `LogsController`'s `getActivityLogs`/`getSecurityLogs`/`getAuditLogs` and their CSV exports applied `tenantId` as a filter only when the caller happened to supply it — omitting it (or supplying a different tenant's id, before the RR-31 guard fix) returned activity/security/audit logs, including audit `before`/`after` payloads, **across all tenants**. | `LogsService` now resolves an `effectiveTenantId` via a new `resolveTenantFilter()` helper: non-super-admin callers always get their own JWT tenant enforced (caller-supplied value is ignored, not just validated); only `super_admin` may pass an explicit tenantId or omit it for an all-tenants view. Applied to all 6 methods. |
| RR-33 | Authorization / Cross-Tenant | `RbacService.updateRole`/`deleteRole` had **no tenant scoping at all** - `findUnique({where:{id}})` with no ownership check. Any caller holding `ROLE_UPDATE`/`ROLE_DELETE` (which `tenant_admin`/`admin` get via `grantAllExcept`) could modify or deactivate **any role by id**, including another tenant's custom roles or shared system roles like `super_admin`/`admin`/`tenant_admin` themselves. | Both methods now take the caller's tenantId + super_admin flag. Non-super-admin callers are rejected (404, not 403, to avoid leaking existence) if the target role `isSystem` or belongs to a different tenant. |
| RR-34 | Authorization / Cross-Tenant | `SsoService.listUserGrants(userId, tenantId)` and `ProductOwnershipService.listForUser(userId, tenantId)` both queried by `userId` alone when `tenantId` was omitted, with no check that the target `userId` actually belongs to the caller's tenant - either endpoint could return another tenant's user's SSO grants / product ownership records just by supplying that user's id. | Both services now validate `userId` is a member of the caller's own tenant (via `UserTenantRole.findFirst`) before returning anything, unless the caller is `super_admin`. 404 on failure, same non-leaking idiom as above. |

All four verified: `tsc --noEmit` exit 0, Jest 1 suite / 11 tests pass, unchanged from before the fix.

**Not fixed (Low severity, same root pattern)**: `FeatureFlagsController.findAll` has the same query-param-tenantId pattern but only returns feature-toggle config, not PII/audit data - now covered by the RR-31 guard fix for the "wrong tenantId supplied" case; the "omitted entirely" case wasn't separately re-verified. Low priority given the data sensitivity; revisit if time allows.

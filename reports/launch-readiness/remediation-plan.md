Deliberately `COALESCE`'d to the literal string `'default'` rather than leaving it `NULL` - Postgres treats every `NULL` as distinct in a unique index, so two null-`applicationId` active subscriptions for the same tenant would silently not collide, defeating the guard for exactly the plans that have no `applicationId` set (a real case in this schema, not a hypothetical). `createSubscription()` now sets `applicationId` explicitly and pre-checks for an existing ACTIVE/TRIALING row (clean `ConflictException`, not a raw DB error, in the non-racing case), with the partial index as the authoritative backstop for the concurrent-request race the pre-check alone can't close - the `create()` call is wrapped in a catch for Prisma's `P2002`/raw `23505` so a race that slips past the pre-check still surfaces as the same `ConflictException` instead of a 500. Confirmed `syncExternalSubscription()` (the other subscription-creation path) is not affected - it already has its own idempotency check keyed on the provider's external subscription ID, a different and already-correct mechanism; left untouched. Verified: `prisma validate` clean, regenerated Prisma Client, `tsc --noEmit` clean, payment-microservice's full Jest suite (21/21, no existing coverage of this module to regress) still passing. **Not verified against a live database** - same sandbox limitation as the rest of this session's Prisma/migration work.
- **RR-04** (`multi-tannet-auth-services`): `login()`'s failure-counter only ever incremented on the "identifier doesn't match any user" path - wrong password and wrong OTP against a real, known account never touched the `login_fail:<identifier>` Redis counter at all, so brute-forcing a real user's password was completely unthrottled by this mechanism (rate limiting from RR-39 still applied at the HTTP layer, but the account-level lockout specifically did nothing). Even on the one path that did count, hitting the threshold only sent a notification email - nothing ever blocked the login attempt itself. Refactored into a shared `recordFailedLoginAttempt()` helper now called from all three failure paths (unknown identifier, wrong password, wrong OTP) and added an upfront lockout check before the user lookup even runs (`UnauthorizedException` if the counter is already at/above 5, checked before any bcrypt compare - avoids wasted work and timing differences on a locked-out identifier). Added a reset (`redis.del`) on successful primary-factor verification, covering all three success exits (bootstrap-password-change, 2FA-pending, full session) so a few earlier mistakes don't keep penalizing a legitimate login. Verified: `tsc --noEmit` clean, IAM Jest suite 11/11 still passing (no existing test covers this method to regress).

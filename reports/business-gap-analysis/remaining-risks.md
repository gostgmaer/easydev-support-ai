- **AI-07 — Cross-tenant circuit breaker grouping — IMPLEMENTED.** `CircuitBreakerManager` (`src/modules/connectors/engine/circuit-breaker-manager.ts`) now keys breakers by `connectorType:baseUrl` instead of `tenantId:connectorId`. Two tenants hitting the same shared upstream (e.g. both on a Stripe connector, or both pointed at the same SaaS host) now share breaker state, so one tenant's failures protect the other from hammering an already-struggling upstream; a tenant-specific baseUrl (e.g. a custom REST connector pointed at that tenant's own backend) still isolates naturally, since the baseUrl itself differs - the regrouping doesn't reintroduce per-tenant noise where tenants are genuinely on different hosts. `ExecutionEngine`'s two `saveBreaker()` call sites and one `getBreaker()` call site updated consistently. Verified: `tsc --noEmit` exit 0, Jest 7 suites / 69 tests pass.

  - **multi-tannet-auth-services**: added `tenant_admin` as a new system role in both seed paths - `permissions-seed.service.ts` (`ensureSystemRolesExist()`'s `systemRoleNames`, runs automatically every boot, additive-only - creates the role row only) and `prisma/seed.ts` (`SYSTEM_ROLES`, the manual data seed that actually assigns permissions) - given the same full-tenant `grantAllExcept` grant `admin` used to carry ("Full access within an assigned tenant"). The bootstrap "admin" user (`BOOTSTRAP_TENANT_ADMIN_EMAIL`/`adminEmail` - the variable was already named `tenantAdminEmail`, which is exactly what it semantically is) is now assigned `tenant_admin` instead of `admin`.
  - `admin` remains a separate, pre-existing role (per the user: "admin is just another role within a tenant with some permissions," not tenant ownership) - **left with its existing full permission grant unchanged** rather than inventing a narrower set with no product guidance on what to remove; flagged in `prisma/seed.ts` with a comment asking for the actual intended permission list if `admin` should be meaningfully smaller than `tenant_admin`.
  - **Verified `super_admin` already has the cross-system/cross-tenant access the user asked to confirm**, across all three services touched this engagement: IAM's own `PermissionsGuard`/`AuthGuard` bypass every permission check and the tenant-isolation check for `super_admin`; `easydev-support-ai`'s `TenantGuard` explicitly exempts `super_admin` from its cross-tenant header check and `RbacGuard` bypasses all role checks; `payment-microservice`'s `PermissionsGuard` does the same. No changes needed - this was already correct everywhere, just unverified until asked.
  - Updated `UsersService.findTenantAdmins(tenantId)` (built below) to query `role.name === 'tenant_admin'` instead of `'admin'`, matching the corrected model.
  - **Live data gap, not a code gap**: any tenant already in a deployed database has no `tenant_admin` row yet - this requires the boot-time seeder to run (automatic, additive) and a real admin user to actually hold that role (today only the manual `prisma/seed.ts`'s bootstrap user gets one automatically). Didn't run `prisma:seed` against any real database as part of this change - that's a deliberate, separate decision the user should make about when to apply it to a live environment.

- **Tenant-admin email lookup — IMPLEMENTED end-to-end, needs one ops step before it's live.** Built across both repos:
  - **multi-tannet-auth-services**: added `UsersService.findTenantAdmins(tenantId)` (queries `UserTenantRole` joined to `Role`/`User`, filtered to `role.name === 'tenant_admin'` and `user.isActive` - see the role-model correction above) and a new `GET /api/v1/iam/users/tenant-admins?tenantId=X` endpoint. Put it in a **new** `TenantAdminsController`, not the existing `UsersController`, because that controller is locked to `AuthGuard` (session-Bearer-only) at the class level and NestJS guards compose rather than override - a route-level guard can't bypass a controller-level one. The new controller reuses `AuthOrApiKeyGuard` (the same dual Bearer-or-`x-api-key` pattern already established by `RbacController`), so a backend job with no end-user session can call it with a global, non-tenant-scoped API key. Registered before `UsersController` in the module's `controllers` array so the static `tenant-admins` path isn't swallowed by `UsersController`'s `GET ':id'`.
  - **easydev-support-ai**: extended the previously-orphaned `IamClient` (`packages/shared-clients/src/iam-client.ts`) with `getTenantAdmins(tenantId)`, calling the endpoint above with an `x-api-key` header. Wired into `UsageLimitService.enforceLimit()` (`src/modules/settings/services/usage-limit.service.ts`): on quota overage, in addition to the existing `admin-incident-job` dashboard entry, now also emails every tenant admin found via the existing `NotificationService.sendEmail()`. Wrapped in the same best-effort try/catch pattern as the incident job - `sendEmail()` throws by design for its queue-worker callers' retry logic, which doesn't apply to this direct call, so failures here are swallowed rather than blocking the enforcement throw.
  - Verified: both repos' `tsc --noEmit` exit 0; `multi-tannet-auth-services` Jest 1 suite / 11 tests pass; `easydev-support-ai` Jest 7 suites / 69 tests pass; `npm run build:packages` (shared-clients) exit 0.
  - **One ops step before this is live**: `AuthOrApiKeyGuard` validates against a real row in IAM's `api_keys` table (`keyHash` = sha256 of the raw key, `permissions` array, nullable `tenantId` for a global/all-tenant key). No such row exists yet for `easydev-support-ai` to use. Needs: generate a key, insert the hashed row with `permissions: ['user:read_all']` and `tenantId: null`, then set `IAM_SERVICE_API_KEY` (raw value) in `easydev-support-ai`'s `.env`. This is a one-time DB/secret provisioning action, not a code gap - didn't fabricate a row in a real database unilaterally, same principle as not inventing secrets elsewhere this engagement.

- **TEN-03 — Billing-plan sync — IMPLEMENTED, entirely within `payment-microservice` + `easydev-support-ai` (no Gateway), per explicit direction to skip the Gateway hop. Code is real, typechecked, and unit-tested; live end-to-end verification is blocked by a pre-existing local Docker/infra issue unrelated to this change (see below).**
  - **payment-microservice** (`src/modules/subscription/subscription.service.ts` / `subscription.controller.ts`): added `findActiveByTenant(tenantId)` - a genuine tenant-only lookup (`findFirst` by `tenantId`, ordered by `createdAt desc`, no `customerId` required, unlike every pre-existing query in this service) - and exposed it as `GET /api/v1/subscriptions/active`, guarded by the same `ServiceOrJwtGuard` + `TenantGuard` + `RequirePermission(SUBSCRIPTION_READ)` as the service's other read routes. Registered before the existing `GET ':id'` route to avoid `'active'` being swallowed as a UUID param. `npx tsc --noEmit` and the existing Jest suite (3 suites / 21 tests) both pass unmodified.
  - **easydev-support-ai** (`packages/shared-clients/src/payment-client.ts`): rewritten from scratch - the old version called a fictional endpoint (`GET /v1/billing/subscriptions/status`) that never existed on the other side and used `Authorization: Bearer`, which isn't the real auth shape either. The new client calls the real `GET /api/v1/subscriptions/active`, authenticates via the legacy `x-api-key` + `x-tenant-id` headers (`ServiceOrJwtGuard`'s documented legacy path), and additionally signs every request with `X-Gateway-HMAC`/`X-Gateway-Timestamp` (HMAC-SHA256 over `METHOD|path|tenantId||timestampMs`, secret `FILE_UPLOAD_HMAC_SECRET`) - discovered live, not from reading code alone, because `GatewayHmacGuard` is a global `APP_GUARD` in payment-microservice that rejects every route, not just the ones the Gateway was assumed to front. Returns `null` (not a fabricated default) on any failure.
  - **easydev-support-ai** (`src/modules/settings/services/usage-limit.service.ts`): `UsageLimitService` now calls the real `PaymentClient`, throttled to once per tenant per hour via Redis (best-effort - if Redis is unreachable it just syncs every call instead of skipping, never the other way around) so this never adds billing-service latency to the hot enforcement path. `Plan.metadata.quotas` (a JSONB field with no migration needed, since the real `Plan` model has no quota columns) is sanitized field-by-field - only finite, non-negative numbers for the known quota keys are accepted - before overwriting any locally-stored limit, since that JSON is owned by a different service's database. No real plan has `metadata.quotas` populated yet, so this is a no-op in production today; it activates automatically the moment billing populates it, no further code change needed. Enforcement (`enforceLimit`, built earlier this pass) remains correct and unaffected either way.
  - Both repos verified clean after the change: `easydev-support-ai` - `tsc --noEmit` exit 0, Jest 7 suites / 69 tests pass. `payment-microservice` - `tsc --noEmit` exit 0, Jest 3 suites / 21 tests pass.
  - **Blocked on live verification, by something unrelated to this change:** a real HTTP call from `easydev-support-ai` to the running `payment-service` container still fails auth, because that container's live `API_KEY_HASH` env var (`d2f333ce9...`, confirmed via `docker exec payment-service printenv`) does not match the value currently in `payment-microservice/.env` on disk (`8c82c302e...`, which *does* match `easydev-support-ai`'s key - the two are supposed to agree). The container is hours older than the current `.env`, and Docker Compose only re-reads `env_file` at container *creation*, not on `restart`. Recreating it (`docker compose up -d --force-recreate`) rebuilds the image successfully but then fails at the network step: the compose file declares `core-network` as `external: true, name: core-network`, but the network actually running on this host is named `easydev-infra_core-network` (a different stack's project-prefixed name) - so the recreate cannot attach to the real Postgres/Redis backing the service. The original container is untouched and still healthy throughout; nothing was forced. This is a pre-existing local-environment naming mismatch between two independently-managed Docker stacks, not something introduced by this change, and fixing it would mean touching network topology shared by other running containers (`core-postgres`, `core-redis`, ...) - outside "entirely within payment service." **Decision (confirmed with the user): stop here.** The code above is the complete, correct implementation; live verification is the only remaining step, and it is gated on infra outside this task's scope.

## Still needs a product/architecture decision (would mean inventing policy, not fixing a bug)

- **TEN-03 — Billing-plan sync (`PaymentClient.getSubscriptionStatus()`).** Investigated against the *real* sibling payment service (`C:\Users\kisho\WorkSpace\Backend\payment-microservice`, confirmed in-workspace, not a true external black box) rather than guessing. Concrete, verified blockers, not speculation:
  - The endpoint `easydev-support-ai`'s client calls, `GET /v1/billing/subscriptions/status`, **does not exist** anywhere in the real service. The integration was built against a contract that was never implemented on the other side.
  - The real `Subscription` model is keyed by `(tenantId, customerId)` - there is no field anywhere in either service's data model mapping "this tenant's billing customer ID." `findByCustomer()` requires both; there's no tenant-only lookup.
  - The real `Plan` model has pricing/billing fields only (`amount`, `currency`, `interval`, `trialDays`, ...) - **no quota fields at all** (no conversations/agents/connectors/AI-request limits). There is nothing to sync quota numbers *from* even if the tenant↔customer mapping existed.
  - Real service-to-service auth is a proper OAuth2 `client_credentials` flow (per-caller JWT issued by IAM, tenant-bound, scope-checked via JWKS) - `easydev-support-ai`'s `.env` only has a bare `PAYMENT_SERVICE_API_KEY` string, which maps to that guard's *legacy* shared-key path, explicitly documented in the payment service's own code as "being retired."
  - **Deeper investigation (with explicit permission to extend the sibling services additively) revealed the real architecture is more involved than "add an endpoint":** IAM's seed data (`multi-tannet-auth-services/prisma/seed.ts`) shows a dedicated **Gateway service** is the architecturally-intended caller for payment operations - it authenticates to payment-microservice with its own `client_credentials` service token (scoped `payment:tenant:any`) and resolves tenant context server-side, rather than each downstream app calling payment directly. A `gateway` container was independently observed running on this host earlier in the engagement, alongside `payment-service`/`auth-service`/`ai-platform-*`, confirming this is a real, separate piece of the platform - not something `easydev-support-ai`'s `PaymentClient` was ever actually wired through. The billing-sync path that *does* exist (a Redis event `events.billing.subscription_status_updated` syncing into IAM's `ProductOwnership` table) is keyed by `(userId, applicationId)`, not tenant, and `ProductOwnership.plan` is a bare string label with no associated quota numbers - confirming quota data doesn't exist anywhere in either real service regardless of which path is used.
  - **Decision (confirmed with the user): stop here, leave as documented.** Integrating properly now means locating and integrating through the Gateway service - a separately-scoped piece of work, not a same-day addition. `UsageLimitService`'s enforcement (built this pass) remains the source of truth for resource limits and doesn't depend on this sync.

- **TEN-04 — Feature-flag gating (`FeatureFlagEngine`).** Unrelated to the payment-service investigation above - `resolveFlag()` is fully internal to `easydev-support-ai` and simply never called by any endpoint guard. Re-checked for an AI-03-style safe default (find an existing, unambiguous mapping and just wire it up) before concluding this needs a decision - there isn't one:
  - The Enterprise-tier flags set at provisioning (`sso.saml`, `audit.export`, `sla.advanced`, `custom.domain`, `white.label`, `priority.support` - `tenant-provisioning.service.ts:267-275`) have **no corresponding endpoint anywhere in this codebase**. There's no SSO/SAML module, no audit-export endpoint, no custom-domain or white-label config surface to gate at all - these flags are aspirational, set for features that don't exist yet.
  - The Growth-tier flags (`workflow.automation`, `analytics.advanced`, `multi.channel`, `connector.custom`, `ai.drafts`, `csat.surveys` - `tenant-provisioning.service.ts:256-264`) are never read by the actual workflow/analytics/connector/AI-draft/CSAT feature code (confirmed by grep - zero matches outside the provisioning service). Every tenant gets identical behavior in all of these areas regardless of plan today.
  - Building enforcement now means inventing two things at once: *which specific actions* within each of these modules should be blocked for an under-tier tenant (the flag names are tier labels, not endpoint maps), and *what denial looks like* (403 with an upgrade prompt vs. 404 hiding the feature entirely). Both are product decisions, not implementation gaps. Not attempted - tell me the specific actions to gate and the desired denial behavior and this becomes buildable immediately.

- **Tenant-admin email lookup (relevant to TEN-03's "overage billing warning" and any future tenant-facing notification).** Investigated the real IAM service (`C:\Users\kisho\WorkSpace\Backend\multi-tannet-auth-services`, confirmed as the actual service `IAM_SERVICE_URL` points at). `GET /api/v1/iam/users?tenantId=X` returns `email` per user but the response **does not include role** at all (`USER_SELECT` in `users.service.ts` has no role/`userTenantRoles` field) - there's no single call to get "the tenant admins, with emails." A new endpoint exposing `UserTenantRole`/`Role` (real models, `role.name` values: `super_admin`/`admin`/`member`/`viewer`/`service_account`) would be additive and low-risk (IAM's schema is young - only one migration ever) and `AuthGuard` already supports service-token callers without a sessionId, so this piece alone was tractable. **Per the user's decision to stop all cross-service work for this pass** (after the Gateway-architecture discovery above), this was not built. Routing quota-overage warnings through the internal `admin-incident-job` dashboard (already shipped) remains the only notification path for now.
- **AI-10 — No review/approval gate before a knowledge document goes live for AI retrieval.** This is a request for a net-new workflow (states, permissions, endpoints), not a fix to existing disconnected wiring like everything else in this pass. Recommended for product backlog.

## Confirmed real, judged not worth the risk/effort relative to value

- **AI-07 — No cross-tenant circuit breaker for a shared upstream connector API.** Confirmed: breaker keys are `connector:{tenantId}:{connectorId}`, so two tenants hitting the same upstream service (e.g. both using a Stripe connector) get independent breakers - each tenant is still protected, just not as fast as a shared breaker would react. Fixing this means grouping breakers by `baseUrl`+`connectorType` instead of tenant-scoped ID, which has its own correctness risk (two tenants at the same hostname might legitimately have different rate limits/regions). Judged: real, but an optimization rather than a correctness bug - not fixed.
- **AI-08 — AI tool-call results aren't linked to a structured business-outcome audit trail.** Confirmed: `AiToolResult.response` stores the raw connector response as opaque JSON. A correct fix needs a structured schema per connector-response-shape (a refund connector's reference ID looks nothing like a different connector's), which requires connector-catalog knowledge this codebase doesn't centralize anywhere. Not fixed.
